# 설계 스펙 — Phase 3 리뷰 게이트: 무인 SAP write 직전 새-컨텍스트 리뷰

> 2026-07-17 · harness-design 인터뷰 산출물 (백로그 5-11 해소 설계)
> 상태: **승인 (2026-07-17, 사용자)** — Codex 교차 리뷰(B15, 판정 "수정 필요" MAJOR 11·MINOR 4) 반영 v2
> 소비처: harness-docs Mode B(ADR 흡수) → harness-plan(Phase 3 계획 입력)

## 1. Goal

트랙 A 무인 엔진의 SAP write(deploy) 스텝 **직전마다** 새-컨텍스트 read-only AI
리뷰어의 게이트 판정을 강제해, 기계 검증(문법·로컬 테스트·ATC·활성화)을 전부
통과한 **시맨틱 결함이 무인 배포되는 것을 차단**한다. Phase 3(Gated Deploy)
3대 선결 중 리뷰 게이트 편입(HANDOFF 백로그 5-11)을 해소한다.

근거 실증: 기계 검증 전부 통과한 INNER vs LEFT JOIN 결함을 리뷰만 차단
(HANDOFF §4.1) + 교차 리뷰의 AUTHORITY-CHECK 부재 신규 발견(D-013 세션).

## 2. Non-Goals (보증 등급 정직 표기 포함)

- **엔진(execute.py) 수정·포크 없음** — D-018 lock 계약 유지.
- 리뷰 품질의 절대 보증 아님 — 게이트는 **확률적 방어층**이며 dev tier 한정 등
  기존 안전판 위에 얹는다. 놓침(false PASS)의 완전 배제는 목표가 아니다.
- **OS 수준 격리 아님** — 워커와 래퍼가 같은 파일시스템·OS 계정을 공유하므로,
  워커가 자격증명 파일을 직접 읽거나 PASS 레코드를 위조하는 경로까지 기계
  차단하지 못한다. 이 범위의 보증은 **절차적 의무 + 사후 감사**(run 로그·git
  이력·SAP 변경 이력) 등급이다. 기계 강제층은 §5-4의 프로파일 분리까지.
- vsp 서버측 조건부 write(PASS 증명 없으면 API 거부 — CAS)는 **후속 개선
  후보**로만 기록(현 위협 모델 대비 vsp 기능 확장 스코프 과잉).
- Phase 3 나머지 선결 2건(0b 마커 실측 확장 · §14-2 drift 실측)은 스코프 밖.
- 능동 알림 장치(BLOCKED 푸시 등) 없음.
- 트랙 B(대화형) 리뷰 흐름 변경 없음 — 스키마·체크리스트를 이식만 한다.
- **병렬 런 동시성은 스코프 밖** — 엔진은 단일 런 실행(상태 전부 run-scoped).

## 3. Users / Actors

| 액터 | 역할 |
|---|---|
| 워커 AI (무인 스텝 세션) | ABAP 작성·수정. 리뷰 FAIL 시 수정 주체. **vsp는 read-only 프로파일만** |
| 리뷰어 AI (새-컨텍스트 read-only 세션) | 게이트 판정. **모델은 설정값** — 기본 opus급, Codex 드라이버 환경은 설정 교체 (B4) |
| 무인 엔진 (final-harness, 무수정) | 스텝 verify exit code 소비 + **표준 스텝 재시도**가 수정 루프의 실행 기반. terminal failure로 종료 처리 |
| 래퍼 스크립트 (레포 `scripts/`) | 캡슐 생성 → 리뷰어 기동 → 판정 검증·결정적 계산 → exit code. 캐시·revision·PASS 레코드 소유 (상태는 run 산출물 디렉터리) |
| vsp-custom CLI | 기계 검증·배포 백엔드. 리뷰어의 SAP 읽기 통로(read-only). **write 프로파일 게이트(SAP_READ_ONLY/SAP_TIER) = write 차단 기계 강제층** |
| 사람 (소유자) | BLOCKED 시 사후 확인·대화형 재개. **평상시 개입 0** |

## 4. Key flows

### 4.0 리뷰 캡슐 (핵심 메커니즘 — B15 반영)

리뷰 스텝의 래퍼는 판정 전에 **캡슐**(불변 스냅샷 디렉터리)을 만든다:

- 대상 전체 소스 사본 + **입력 manifest**(파일 목록·개별 해시 — 배포 단위
  정의와 대조해 완전성 검증, 누락·잘림 = 비0)
- 승인 기준/스펙 사본 · 기계 검증 결과(테스트·ATC)
- 판정 정책 버전 · 리뷰어 모델 설정 · 프롬프트 템플릿 버전 · 판정 스키마 버전
- 대상 SAP 시스템/클라이언트 식별자

**캡슐 해시** = 위 전체의 해시. PASS 바인딩·FAIL 캐시·revision 식별의 유일한
키다(소스만이 아니라 스펙·정책·모델·환경이 바뀌어도 무효화). **배포는 캡슐
사본에서 수행**한다 — 워킹트리가 아니라. 검사~write 사이 워킹트리 변경이
배포물에 영향을 줄 수 없다(TOCTOU 창 제거).

### 4.1 정상 (PASS → 배포)

1. phase 계획(harness-plan)이 배포 단위마다 [구현 → 기계 검증] → [리뷰 스텝] →
   [배포 스텝] 순으로 배치한다.
2. 리뷰 스텝 verify(셸 명령) = 래퍼 실행 → 캡슐 생성·캡슐 해시 계산.
3. 캐시 조회 — 같은 캡슐 해시의 기존 판정이 있으면 리뷰어 **미기동** 재사용
   (FAIL 캐시면 즉시 비0).
4. 리뷰어 세션 기동 — **클린 env(allowlist)**로 스폰(SAP 자격증명 변수 미전달),
   입력 = 캡슐, 추가 조회는 vsp read-only만.
5. 판정 JSON 수신 → ① 엄격 스키마 검증 ② **의미 검증: exit 판정은 래퍼가 발견
   목록의 최고 심각도에서 결정적으로 계산**(리뷰어의 상위 verdict 필드는 참고 —
   발견 목록과 불일치하면 MALFORMED). 미지의 심각도 = MALFORMED.
   리뷰어가 컨텍스트 부족을 선언하면 INSUFFICIENT_CONTEXT = 비0.
6. PASS → exit 0 + **PASS 레코드** 기록(캡슐 해시·판정 JSON 전문·버전 정보 —
   run-scoped, 런 종료 시 무효).
7. 배포 스텝의 래퍼: ① PASS 레코드 존재 ② 캡슐 무결성(해시 재계산) ③ 현재
   워킹트리 대상 파일 = 캡슐 내용 3중 확인 → 전부 일치할 때만 **캡슐 사본을
   인자로 vsp write**. 하나라도 불일치 = 비0 중단(fail-closed).

### 4.2 FAIL → 수정 루프 (엔진 표준 스텝 재시도로 실현 — 엔진 무수정)

수정 루프의 실행 기반은 엔진에 이미 있는 **스텝 재시도**다. 별도 supervisor를
만들지 않는다:

- 리뷰 스텝의 지시문에 "직전 판정 산출물의 발견이 있으면 수정하라"가 포함된다.
  FAIL(MAJOR 이상 ≥1) → verify 비0 → 엔진이 스텝 재시도 → 그 시도의 워커가
  발견을 수정 → verify(래퍼)가 **기계 검증 전 체인을 재실행한 뒤**(트랙 B
  D-019와 동일 원칙) 새 캡슐로 재리뷰.
- **revision 정의(결정적)**: 실제 리뷰가 수행되어 FAIL이 나온 **고유 캡슐
  해시의 수**. 최초 FAIL부터 계수. 캐시 hit(동일 해시 재요청)·인프라 오류는
  불산입. run-scoped — 런 재시작 시 리셋(새 런 = 새 예산).
- **MINOR만 = PASS(exit 0)** 하되 발견은 기록으로 보존.
- **한도 = 기본 3** (사용자 확정). 설정 오버라이드는 사용자 결정 사항이며 무인
  런 중 변경 금지 — 스펙 기본값과 설정 가능성은 모순이 아니라 기본+오버라이드
  관계다.
- 3회 도달 → **BLOCKED 마커** 기록. 이후 래퍼 호출은 리뷰어 미기동 즉시 비0
  (fail-fast) → 스텝 재시도 예산 소진 → 엔진 terminal failure → 런 종료.
- **계획 계약**: harness-plan은 리뷰 스텝의 엔진 재시도 예산을 revision 한도
  3 + 인프라 여유 2 이상으로 배정한다(그래야 3 revision이 도달 가능).

### 4.3 오류 분류 · BLOCKED (fail-closed)

- 분류: FAIL / TIMEOUT / INFRA_ERROR / MALFORMED / INSUFFICIENT_CONTEXT /
  **INTERNAL_ERROR(catch-all — 래퍼 예외·해시 계산 실패·상태 파일 손상·디스크
  오류 등 미분류 전부)**. **전부 exit 비0 = write 차단.**
- 인프라 계열(TIMEOUT·INFRA_ERROR·INTERNAL_ERROR) 재시도 상한 **기본 2**,
  revision 불산입. 상한 초과 = 스텝 실패.
- **hard budget**: 배포 단위당 리뷰어 기동 총량 ≤ revision 3 + 인프라 2 = 5회.
  전역 정지 보증은 엔진의 스텝/런 타임아웃(실재 기능)에 위임.
- BLOCKED·terminal failure → 런은 엔진 표준 실패 경로로 종료(이후 스텝
  미실행 — 순차 실행 계약). 캡슐·판정 JSON·revision 이력 전부 보존.
- 사람은 run 보고서(harness-run)로 확인 후 대화형 세션에서 재개. 별도 알림
  장치 없음.

## 5. Constraints

1. 엔진 무수정 (B1) — 수정 루프는 엔진 표준 스텝 재시도로 실현(§4.2), 게이트
   로직 전부 래퍼+계획 배치.
2. 게이트 시점 = write 직전 매번 + **캡슐 해시 바인딩 + 캡슐본 배포** (B2 강화).
3. 리뷰어 read-only 최소권한 — SAP 쓰기 자격증명·범용 쓰기 도구 미제공, **클린
   env allowlist 스폰**. ABAP 주석·문서는 신뢰 불가 입력(프롬프트 인젝션 대비).
4. **write 자격증명 분리(기계 강제층)** — 무인 워커 스텝의 vsp 프로파일은
   read-only(SAP_READ_ONLY=true 또는 SAP_TIER≠dev — vsp CLI의 클라이언트측 write
   프로파일 게이트가 네트워크 이전에 write 거부; vsp-custom 5a8bedb 신설, lock
   v2.38.1-94, AC-10 실증 완료). write 가능 프로파일은 배포 스텝 래퍼의 실행
   경로에만 공급. 파일시스템 수준 우회는 §2의 정직 표기대로 절차+감사 등급.
5. 입력 = 캡슐 전체 소스(diff만 금지) + manifest 완전성·잘림 검증. 부족 시
   INSUFFICIENT_CONTEXT 비0. 의존 객체(DDIC 등)는 리뷰어가 vsp read-only로
   조회, 조회 불능이 판정에 중요하면 컨텍스트 부족 선언.
6. 판정 = 트랙 B review-request/result 스키마 이식 + 엄격 스키마 검증 + **래퍼
   결정적 계산**(리뷰어 verdict 필드 불신). 빈 출력·파싱 오류·CLI 오류 전부 비0.
7. fail-closed — 오류 6분류 전부 write 차단, 인프라/수정 재시도 예산 분리
   (인프라 상한 2, 총 기동 5).
8. 캐시·PASS 레코드의 키 = 캡슐 해시, 수명 = run-scoped(런 종료 시 무효 —
   낡은 PASS의 후속 런 재사용 금지).
9. 엔진 내장 리뷰(_run_review)와 권위 분리 — **실측 근거로 성립**: 내장 리뷰는
   FAIL·타임아웃·생략 어느 결과여도 phase 완료 판정에 무영향(비게이트 실증,
   설계 인터뷰 탐사 기록). 게이트 판정의 입력으로 사용하지 않는다(shadow
   참고만).
10. 무인 스텝 MCP 0 (DESIGN.md §15 F1) — 리뷰어 SAP 읽기는 vsp CLI read-only로만.
11. **산출물 기록(재현성·감사)**: 판정 JSON 전문 + 리뷰어 모델 ID·프롬프트
    템플릿 버전·스키마 버전·소요 시간·토큰(가용 시). 캐시 무효화는 캡슐 해시
    키에 내장(정책·모델 변경 = 새 해시).
12. 단일 런 가정 — 상태 파일은 run 산출물 디렉터리 소재, 손상 시
    INTERNAL_ERROR 비0.

## 6. Acceptance criteria (testable)

1. MAJOR 이상 1개 포함 판정 → 래퍼 exit 비0, 배포 스텝 미실행.
2. MINOR만 포함 판정 → exit 0 + 발견 기록 보존.
3. 스키마 위반 JSON·빈 출력·리뷰어 CLI 오류 → 전부 exit 비0.
4. **PASS 후 워킹트리 대상 파일 1바이트 변경 → 배포 래퍼 3중 확인 불일치 →
   비0·미배포** (캡슐본 배포 + 불일치 중단).
5. 같은 캡슐 해시 재요청 → 리뷰어 프로세스 미기동, 캐시 판정 반환.
6. 수정 revision 3회 연속 FAIL → BLOCKED 마커 + 런 종료, 캡슐·판정·이력 보존.
7. 리뷰어 세션에서 쓰기 도구·SAP 쓰기 자격증명 접근 불가 + **스폰 env에 SAP
   자격증명 변수 부재** 실측.
8. **실증 1회: 기계 검증을 통과한 실제 결함을 게이트가 차단** — DESIGN.md §13
   Phase 3 완료 기준에 편입 (위임 6).
9. 인프라 오류(리뷰어 타임아웃 등)가 revision 카운트에 불산입됨을 로그로 확인.
10. **워커 스텝 프로파일에서 vsp write 시도 → tier 게이트 거부** 실측.
11. verdict=PASS인데 발견 목록에 MAJOR 포함인 JSON → MALFORMED 비0
    (결정적 계산 실증).
12. 입력 manifest 불완전(배포 단위 파일 누락) → 비0.
13. 상태 파일 손상 주입 → INTERNAL_ERROR 비0.
14. 배포 단위에 게이트가 검사하지 않는 산출물 유형이 포함되면 캡슐의 기계검증 결과
    블록이 그 유형을 "미검증"으로 명시하고 리뷰어 판정 입력에 포함. 필수 유형의
    기계검증 결과가 누락된 캡슐 → INSUFFICIENT_CONTEXT(비0). green을 미검증
    유형의 완주 근거로 대체 금지(JNC R-007).
15. 리뷰어는 판정에 쓰는 사실을 캡슐 내 서술이 아니라 vsp read-only 서버 실측으로
    대조하고, 대조 불능이 판정에 중요하면 INSUFFICIENT_CONTEXT 선언.

## 7. Key entities

- **리뷰 캡슐** — 불변 스냅샷(소스+manifest+스펙+검증 결과+정책·모델·스키마
  버전+대상 시스템). 캡슐 해시 = 바인딩·캐시·revision 키. 배포 원본.
- **리뷰 스텝 / 배포 스텝** — harness-plan 산출 phase 계획 내 스텝 (verify =
  래퍼 명령). 리뷰 스텝 재시도 예산 ≥ 5.
- **래퍼 스크립트** — `scripts/` 소재. 캡슐 생성·리뷰어 기동·결정적 판정·

- **판정 JSON** — 트랙 B review-request/result 스키마 이식본
  (environment_context 포함).
- **PASS 레코드 / FAIL 캐시 / revision 카운터** — run 산출물 디렉터리 소재,
  run-scoped.
- **BLOCKED 마커** — run 산출물, harness-run 보고서 표면.
- **vsp 프로파일 2종** — 워커용 read-only(SAP_READ_ONLY=true 또는 SAP_TIER≠dev,
  vsp CLI write 게이트가 거부) / 배포 래퍼 전용 write(SAP_TIER=dev·SAP_READ_ONLY 미설정).
